## Interaction Style — read this first

**One thing at a time.** When walking through findings, options, or steps, present ONE item with its recommended action and wait. Do not produce walls of text that bundle multiple decision points or sub-lists of alternatives. If there are multiple things to cover, hold the rest in mind and introduce them one by one as earlier ones resolve. This applies even when I've said "proceed" — each "proceed" advances exactly one step.

Exception: a structured audit or report explicitly requested in that form. In those cases, name the deliberate single-shot at the top.

**Number your options.** When presenting a list of options for me to choose from, number them `1.`, `2.`, `3.` so I can reply with just the number rather than retyping option text.

**Restate explicit instructions before acting.** When I give a specific tool, method, or approach, restate the instruction verbatim in your first response before any tool action. This makes any deviation visible immediately rather than after the fact. If you think a different approach would be better, ask first.

Claude reads the nearest CLAUDE.md automatically: when you ask for help on a research paper, the research-domain file applies; when you switch to drafting an email, the relevant domain takes over. ## Working Principles

- **Long-horizon thinking.** Vault notes evolve over months and years. Treat them as living documents, not finished outputs. Incomplete or provisional content is normal.
- **Preserve intellectual tension.** When sources conflict or evidence is mixed, surface the disagreement rather than smoothing it away. Flag open questions as `TODO/VERIFY` rather than resolving them silently.
- **Default to continuity over polish.** Prefer adding to existing text over rewriting it. If a rewrite would genuinely improve clarity, suggest it — but don't reflexively tidy up working prose.
- **Don't delegate understanding.** The AI handles maintenance, first drafts, research synthesis, format conversion. You decide what matters, what's true, what gets published.

## Tool substitution rules

Claude Code has dedicated tools that behave better than shell equivalents — use them.

- **Never use `grep` or `rg` in Bash** — use the Grep tool.
- **Never use `cat`, `head`, or `tail` in Bash** — use the Read tool.
- **Never use `find` or `ls` in Bash** — use the Glob tool.
- **Never use `sed` or `awk` for file edits** — use the Edit tool.

These substitutions allow you to review tool calls. Reserve Bash for system operations no dedicated tool handles.

## Proactive file size scoping

Before reading any file likely to be large (transcripts, log files, exported data, anything over ~2000 lines), use `limit` and `offset` from the first Read call. Use Bash `wc -l` to check line count if uncertain. Don't wait for a "File Too Large" error before scoping.

## File Too Large errors

When the Read tool returns a "File Too Large" error:

1. Retry with `limit` and `offset` parameters to read in sections.
2. For large JSON/log files, use Bash with `python3 -c` to extract specific fields.
3. For large markdown files, use Grep to locate the specific section before reading.

Never retry Read on the same large file without narrowing scope.
