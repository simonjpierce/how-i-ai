# Lesson 6 — Your first real task

You now have a working setup. Time to use it for something real.

> **Video** *(placeholder):* `<https://loom.com/...>` — me picking one task from my own pile and doing it from scratch with the new setup, ~3–4 minutes.

## Pick something real

The fastest way to internalise this system is to use it on actual work, not a toy exercise. So pick something:

- A meeting recording you've been meaning to transcribe and process.
- A grant section that's overdue for a first draft.
- A PDF you should have read three weeks ago.
- A report you've been putting off.
- A draft email to a funder, donor, or collaborator.
- A research question you'd normally Google.

If you're stuck choosing, pick the smallest item on the list above that produces a deliverable. Done > perfect.

## Prompt seeds by track

Just to give you a starting point — pick the line that fits your work and paste it into Claude Code with the relevant file or context attached.

**Writers (anything long-form)**

```
Read [the project note / the meeting transcript / this Word doc] and draft
[a 200-word funder update / an email to the team / the introduction section].
Match the voice in [voice reference / a past draft I'm pointing you at].
Ask me a clarifying question first if anything's missing.
```

**Researchers**

```
Read [the project notes for X] and tell me what's the current state of play —
who's writing what, what's outstanding, what decisions are open.
Then suggest the highest-leverage next step.
```

**Analysts (anyone working with data)**

```
Here's a CSV / spreadsheet. Read it, tell me what's in it, and propose
2–3 things we could learn from it. Don't run anything yet — just propose.
```

**Communicators / managers**

```
I'm about to email [person] about [topic]. Read [the relevant project notes /
past correspondence] and draft a short message. Conversational tone.
```

These are seeds, not scripts. The point is to drop yourself into a real workflow, not to follow a template.

## What to watch for

- **Claude narrates what it's doing.** It'll say "I'm going to read your project notes first." That's useful — if it's about to do the wrong thing, interrupt with Ctrl-C and redirect.
- **You can interrupt at any point.** Nothing is permanent until Claude finishes a tool call. If a draft is going sideways, say so: *"Stop. Actually I want X instead of Y."*
- **Output goes somewhere specific.** When Claude writes a file, it tells you the path. Open it in Obsidian and edit there — the file is yours now, not Claude's.

## When you finish

Two skills are worth running *every* session after a real piece of work — they're covered in lessons 7 and 8 of the course, but you can use them from day one:

- **`/document`** — wrap up the session by recording what was done, what's open, and what's next. Tomorrow's session reads that file first.
- **`/update`** — sweep related project notes and docs for things that should be updated based on what just happened.

Run those, then close the window. You've used the system.
